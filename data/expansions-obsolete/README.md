# Expansions

Each `.txt` here is a named **expansion**: a fixed snippet of prompt text that is spliced
in verbatim wherever you write `<name>`. Unlike a list (which draws one random entry),
an expansion always inserts the **same** text — though that text may itself contain
dynamic prompts (`#name`) and lists (`{name}`), which are then expanded, so an expansion
can still introduce randomness.

## How to reference an expansion in a prompt

Write `<name>` in a prompt. The name is resolved against every expansion file by **path
suffix** — the same rule lists use — so all of these work and you almost never type the
folder:

- **Bare filename** — `<rays>` → `lighting/rays`
- **Short / partial path** — `<lighting/rays>`
- **Full path** — `<lighting/rays>`

Resolution is deterministic: an exact path wins; otherwise any file whose path ends with
`/<your-ref>` matches; among matches the **shallowest** path wins, ties broken by a
guaranteed natural order (symbols, then numbers, then letters). Folders can nest as deep
as you like; the suffix lookup keeps references short. Basenames are kept unique so a
bare `<name>` always resolves.

Expansions expand recursively (up to 10 passes) and are **LoRA-safe** — `<lora:name:weight>`
is masked so it is never treated as an expansion.

## Folders

Expansions are grouped into category folders. A folder with 2+ expansions is also a
**pick-one group**: `<folder>` splices ONE random expansion from it (e.g. `<lighting>` →
a random lighting snippet). It picks a single member and splices it — never a union of all
members' text:

| Folder | What's in it |
|--------|--------------|
| `detail/` | Detail & quality boosters — `legacy`, `legacy-person`. Marked `_force-prefix`, so these show/insert as `<detail/legacy>` (the folder supplies the "detail" context) |
| `style/` | Rendering / art-site style tags — `pixelart`, `dap` |
| `lighting/` | Lighting and atmosphere — `rays`, `candlelight` |
| `subject/` | Subject snippets — `coffecup`, `flower-pic` |
| `scene/` | Scene / setting snippets — `underwater-anime-irl` |

## Metadata (`<name>.json`)

Each expansion may have an optional `<name>.json` sidecar next to it (e.g.
`lighting/rays.json`) holding metadata — currently just a `description` used for the
editor button tooltip:

```json
{ "description": "God rays — light shafts and volumetric lighting." }
```

A category folder may also carry a `<folder>.json` (e.g. `lighting.json`) whose
description becomes the tooltip on that category's pill in the editor. The project ships
one for every built-in expansion and folder. Metadata files are never themselves
expansions — never read as snippets, never shown as buttons. They're regenerated by
`scripts/expansion-meta/write-expansion-meta.mjs`.

## Internal / config files (`_` prefix)

Any file whose name starts with an underscore is **internal/config**, never an
expansion — the loaders ignore `_`-prefixed files entirely (the same convention as
`data/lists/`). They are plain files, not dotfiles, so Vite's `import.meta.glob` can see
them in the SPA build.

A folder containing an empty **`_force-prefix`** file shows/inserts its entries with the
path from that folder down (e.g. `detail/` has one, so `legacy` shows as `<detail/legacy>`).
This lets a short basename keep its folder for context instead of standing alone — the
reference still resolves by suffix as usual, so the prefix is shown, not strictly required.

## Content

Expansions are small, hand-authored snippets (not generated from the CSV sources), so
they carry no slurs / extreme / minor-sexualizing content and need no runtime filtering.
Keep new ones in the same spirit; an adult snippet would follow the same `nsfw`
name-token convention lists use.

See `../lists/README.md` for the parallel list system and
`../../notes/reference/expansions-architecture.md` for the design.
